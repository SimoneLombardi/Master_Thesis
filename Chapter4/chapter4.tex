\section{Unified Robot Architecture: JointRobotTP Class}
This section explains the implementation of the \textit{unified robot architecture} that is mainly contained in the node: \textit{joint\_robot\_tp\_node}, that is the fundation of the control nodes, as shown here \ref{fig:simulation_nodes_diag} .\\
\noindent All the code was developed in the \codetxt{darkgray}{c++} language, for performance, and the matrix computation is handled by the library \textit{Eigen}.\\
\noindent Here is a simplified diagram of the main class developed to manage the control of the robotic system:\\
\begin{figure}[H]
	\centering
	\includegraphics[scale=0.5]{simplified_class_structure.png}
	\caption{Simplified structure of the class JointRobotTP}
	\label{fig:JointRobotTP simplified}
\end{figure}
\noindent As shown in \ref{fig:simulation_nodes_diag}, there are three control nodes; the two nodes for the robots are members of the  \codetxt{darkgray}{JointRobotTP} class.
They were developed during the \textit{Sesto senso project} and contain the data structure for the following application:
\begin{itemize}
	\item \textit{Transformation} listening and broadcasting, using the \textit{tf2\_ros} pkg.
	\item \textit{Inverse kinematics} using using the \textit{KDL} pkg to recursively compute the jacobians of the robots. Starting from the \textit{urdf} description of the robots.
\end{itemize}
\noindent
The transformation buffer contains the kinematic chain of the two robots, as \codetxt{darkgray}{geometry\_msgs::msg::TransformStamped} messages. These transformations are periodically updated through a topic, published by the \textit{robot state publisher}.
The \textit{tf2\_ros} package also allows to retrieve specific frame-to-frame transformation, using the frames id, defined in the \textit{urdf} files.\\

\noindent The main goals I wanted to pursue during development were the following:

\begin{table}[ht]
	\centering
	\begin{enumerate}
		\item Have an efficient way to send commands to the unified robot. \label{obj:comm_efficiency}
		\item Use a data structure that allows for easy addition/removal of tasks. \label{obj:task_flexibility}
		\item Keep a degree of separation between robot info and control. \label{obj:decoupling}
	\end{enumerate}
	\caption{Code structural objectives for the manipulator control system.}
	\label{tab:code_objectives}
\end{table}

\noindent In the following section, the structure of the \textit{JointRobotTP} class will be described in more depth.

\subsection{Class Initialization}
The \textit{Initialize} method is called at start-up to instantiate all the members of the class object, data structure, publishers, subscriber and the action server.
\begin{figure}[H]
	\centering
	\includegraphics[scale=0.5]{initialize_diagram.png}
	\caption{activity diagram of the \codetxt{olive}{initialize} method, \ref{chap:appendixA}}
	\label{fig:initialize_diagram}
\end{figure}
\noindent In the first step, the robots' nodes \textit{kuka\_robot\_node} and \textit{ur10\_robot\_node} are created and initialized with their class-specific method, used for interface configuration and internal state initialization.\\
\noindent Next the communication interfaces are created, the \textit{proximity task} subscriber used to recive data from the \textit{Environment handling} stak \ref{fig:simulation_nodes_diag}. And the \textit{Action server}, used to send \textit{goals} in \textit{position} and \textit{orientation} to the unified robot. \\
\noindent For the last step, the method handles the initialization of a series of variables and data structures used later in the node.
\begin{itemize}
	\item \textbf{Parameters}: all the gains, threshold, and other run-time variables are created as \textit{ros2 parameters}.
	\item \textbf{Initial joint configuration}: stored as a \codetxt{darkgray}{std::map}. The method populates a list of vectors containing potential initial configurations.
	\item \textbf{Task update function}: the method stores the pointers to the update of the active \textit{tasks}.
\end{itemize}
\noindent
This structure was chosen to have a more manageable code and to accomplish the second objective, as mentioned in \ref{tab:code_objectives}, set for the architecture.

\subsection{Task state update}
Just to explain the logic in the implementation, for each task defined for a particular objective in the control algorithm, three different methods were developed, used to cyclically update the information contained in the task \textit{data structure}.\\
The pointers to these methods are inserted in a vector, and this vector is used to call all the functions in a loop. This is functional since it is possible to remove a task from the update cycle just by skipping a cycle in the loop, and this leaves a clear and concise way to show which task are being updated, as for point two in \ref{enum:StructuralObjective}.

\subsection{Task data structure}
For the control algorithm, three different matrices are defined for each task in the hierarchy. A custom \textit{stuct} was implemented to store the matrices and each task was saved in a map(\codetxt{darkgray}{map<string, tp\_task>} \codetxt{cyan}{TP\_task\_map\_}\codetxt{darkgray}{;}), to allow referencing each task by name, avoiding numeric index that are not explanatory enough.\\[6pt]
\noindent Saind \textit{struct} is defined as follows:
\\[6pt]
\noindent\makebox[\textwidth]{
	\begin{tabular}{l}
		\codetxt{darkgray}{Eigen::MatrixXd} \codetxt{cyan}{RefRate}\codetxt{darkgray}{;}\\[4pt]
		\codetxt{darkgray}{Eigen::MatrixXd} \codetxt{cyan}{ActMatrix}\codetxt{darkgray}{;}\\[4pt]
		\codetxt{darkgray}{Eigen::MatrixXd} \codetxt{cyan}{TskJacobian}\codetxt{darkgray}{;}
	\end{tabular}
}
\noindent This definition uses dynamically sized matrices that are useful in the case of the \textbf{task priority} control since the dimension of each matrix can change from task to task, maintaining a certain relation within each task:
\begin{equation}
	\begin{aligned}
		\mathrm{RefRate}     &\in \mathbb{R}^{(m\times1)} \\
		\mathrm{ActMatrix}   &\in \mathbb{R}^{(m\times m)} \\
		\mathrm{TskJacobian} &\in \mathbb{R}^{(m\times n)}
	\end{aligned}
\end{equation}
\noindent The use of \codetxt{darkgray}{std::map} is also computationally efficient(search complexity $O(\log n)$, with n the element number in the map).


\section{Unified Robot Architecture: Action server}
The action server in the \textit{ros2} framework is composed of various methods, used for the \textit{acceptance}, \textit{rejection}, and \textit{abortion} of goals sent to the server. The main body of the server instead is located in two methods called: \codetxt{olive}{execute}, and \codetxt{olive}{RunCartesianReachingLoop} also found here \ref{chap:appendixA}.

\subsection{Execute method}
This method is executed in a separate thread, created after the goal has been accepted. Firstly, from the \textit{goal\_handle} message, the \textit{initial joint configuration code} is retrived and a simple control loop is then used to control the joint:
\begin{figure}[H]
	\centering
	\includegraphics[scale=0.4]{JointControlLoopDiagram.png}
	\caption{Joint reaching control loop}
	\label{fig:JointControlLoop_flowchart}
\end{figure}
\noindent
The loop drives each joint towards the desired configuration. The integration of this part in my architecture was done to improve repeatability, to have the possibility to easily set a common starting position in different experiments.\\
\noindent The second procedure is the projection of the \textit{requested goal} in the correct \textit{reference frame}, and after that, said goal is broadcast in RViz using a publisher.

\section{Unified Robot Architecture: Control algorithm}
% TP computation class description
The structure of the control algorithm was adapted to the \textit{task state} data structure. Two main objective were considered in the code development:
\begin{itemize}
	\item \textit{Passive priority}: removing the necessity to use a flag, improving code readability.
	\item \textit{Easy task integration}: easy task insertion in the hierarchy.
\end{itemize}
\noindent With this in mind a class called: \codetxt{darkgray}{TPComputation} was created, of witch an object could be instantiated for each iteration of the reaching loop, the class data would be the \textit{null space} of the \textit{task jacobian} computed at the previous step, $Q_{k-1}$, and the resulting \textit{joint velocity} vector, $\rho_{k-1}$. These data would be extracted from the class as needed to be sent to the robot. To initialize the class object the method \codetxt{olive}{init\_TPComputation} takes as arguments the number of \textit{Dof} of the structure, used to initialize the dimension of the internal matrices, and some parameters that are used in the pseudo-inversion of the \textit{augmented jacobian}.
Since I wanted to be sure that no information remained after any cycle, a method to remove all the old data was created. That \textit{cleared} the internal variables after use. The step computation of the \textit{task priority inverse kinematics} is performed using the \codetxt{olive}{computeTP\_step} \codetxt{darkgray} method. The arguments taken by the function are the matrices of the desired task, stored in \codetxt{darkgray}{map<string, tp\_task>} \codetxt{cyan}{TP\_task\_map\_}\codetxt{darkgray}{;}. Internally, the equations \ref{equat:Modern_TP_algorithm} and \ref{equat:RegPseudoInverse_operator} are implemented. Lastly, after all the desired steps have been computed, the resulting \textit{joint velocity vector} is extracted using: \codetxt{olive}{getTP\_ydot}.
\noindent This implementation is done in relation to point three in \ref{tab:code_objectives}.


\subsection{Reaching loop process flow}
To better show the intended and actual use of the \codetxt{darkgray}{TPComputation} class to control the robot, I added the flow-chart for the \codetxt{olive}{RunCartesianReachingLoop} method. 
\begin{figure}[H]
	\centering
	\includegraphics[scale=0.4]{Reaching_loop_flowChart.png}
	\vspace{-60pt}
	\caption{Reaching loop flow chart}
	\label{fig:ReachingLoop_flowchart}
\end{figure}
\noindent To briefly describe the loop, after the instantiation of the \codetxt{darkgray}{TPComputation} object, the loop starts.  In the \textit{Update step}, all the functions that are relative to a \textit{task} state are called, starting with the \textit{Reference Rate}, \textit{Activation function}, and lastly \textit{Task Jacobian}.
In the second part, the \textit{Stop condition} is checked. If the control is positive, the loop is immediately stopped, and the result is sent to the action server client.
If the opposite is true, the \textit{tp\_controller} is initialized, with the \textit{Dof} of the system and the variables for the pseudo-inverse computation. Than each "\textit{step}" of the algorithm is performed, finishing with a "\textit{null}" task, composed of two Identity matrices for \textit{Activation function} and \textit{Task Jacobian}, and with a zero vector for \textit{Reference Rate}.
The loop is repeated until the stop condition is met or stopped after a time limit; in this case, the goal is considered not reached.
